# 测试报告

## 一、执行摘要

本项目同时执行离线 pytest 回归测试和真实 LLM 在线质量评测。

| 指标 | 结果 | 结论 |
| --- | ---: | --- |
| 离线回归测试通过率 | 100%（7/7） | 执行器与数据生成器逻辑稳定 |
| 首轮场景通过率 | 75%（3/4） | 混合日期首次生成失败 |
| 代码可执行率 | 100%（4/4） | 四个场景生成的最终代码均可执行 |
| 数据值正确率 | 100%（4/4） | 最终输出值全部符合预期 |
| 自修复后通过率 | 100%（4/4） | 失败反馈后混合日期修复成功 |

总体结论：模型首轮并非稳定全对，但执行反馈闭环能够修复本轮出现的日期解析错误。相比单独报告 7 passed，这组在线结果直接反映了生成代码的可执行性、数据正确性和自修复效果。

## 二、评测环境与方法

- 评测日期：2026-06-24。
- 模型：通过 DeepSeek 兼容 API 调用 deepseek-chat。
- 温度：0.1。
- 在线场景数：4。
- 最大尝试次数：3。
- 执行方式：模型生成 clean_data 函数，由项目 executor.py 在 DataFrame 副本上执行。
- 判定方式：检查异常、返回类型、行数、字段名和逐项期望值。
- 原始结果：evaluation/results.json。
- API 密钥仅从进程环境读取，不写入仓库。

## 三、指标定义

- 首轮场景通过率：第一次生成代码即通过全部检查的场景数 / 总场景数。
- 代码可执行率：最终代码能够执行并返回 pandas DataFrame 的场景数 / 总场景数。
- 数据值正确率：最终结果逐项符合预期值的场景数 / 总场景数。
- 自修复后通过率：最多三次生成后通过全部检查的场景数 / 总场景数。
- 场景通过条件：代码可执行、行数保持、字段契约正确、数据值正确，四项必须同时满足。

## 四、在线场景明细

| 场景 | 输入重点 | 首轮 | 尝试次数 | 最终值 | 结果 |
| --- | --- | ---: | ---: | ---: | ---: |
| 价格清洗 | 美元符号、USD、FREE、千位逗号 | 通过 | 1 | 正确 | 通过 |
| 日期标准化 | ISO、英文日期、无效日期、斜杠日期 | 失败 | 2 | 正确 | 通过 |
| 嵌套 JSON 展开 | 正常 JSON、无效 JSON、null | 通过 | 1 | 正确 | 通过 |
| 用户编号规范化 | U1001、user-1002、纯数字、空值 | 通过 | 1 | 正确 | 通过 |

### 价格清洗

输入：

- $19.99
- USD 20
- FREE
- 1,000.50

期望输出 price：

- 19.99
- 20.0
- 0.0
- 1000.5

模型首轮正确去除货币符号和千位分隔符，并将 FREE 映射为 0.0。字段、行数和数据值全部通过。

### 日期标准化与自修复

输入：

- 2023-10-01
- Oct 2nd 2023
- invalid_date
- 2023/10/03

期望输出：

- 2023-10-01
- 2023-10-02
- NaT
- 2023-10-03

首轮代码未正确处理混合格式日期，因此数值检查失败。评测器把期望值、实际值和失败状态反馈给模型。第二次生成改为逐值解析：有效日期转换为 Timestamp，无效日期转换为 NaT。最终四项值全部正确。

这一结果验证了自修复闭环确实生效，但也表明首轮生成对 pandas 混合日期格式并不可靠。

### 嵌套 JSON 展开

输入包含两条有效 JSON 和一条无效 JSON。要求生成：

- metadata_category
- metadata_warranty_years

模型首轮成功解析有效 JSON，将无效 JSON 转为缺失值，并删除原 metadata 字段。字段契约和数据值均通过。

### 用户编号规范化

输入包括 U1001、user-1002、1003、U-1004 和空值。期望统一为 U-数字 格式，空值保持缺失。模型首轮正确提取数字并完成格式统一。

## 五、离线 pytest 结果

pytest 共 7 项，全部通过：

- 清洗脚本返回 DataFrame 且不修改原始输入。
- 缺少 clean_data 或返回错误类型时正确报错。
- 原始异常类型、消息和 traceback 能被保留。
- 动态代码环境提供 pandas 和 json。
- 销售数据生成器输出正确字段和行数。
- metadata 合法记录满足字段契约。
- 日志生成器输出 raw_log 并符合预期格式。

离线测试用于保护执行边界；在线评测用于衡量真实模型生成质量，两者不能互相替代。

## 六、风险与改进建议

- 首轮成功率只有 75%，生产流程必须保留数据质量验证和重试。
- 生成代码使用 exec 执行，目前只限制了命名空间，并不是完整安全沙箱；不应直接运行不可信模型代码。
- 当前仅有 4 个在线场景，建议扩充到至少 20 个，加入空 DataFrame、重复值、时区、超大金额、Unicode、日志缺字段和恶意代码提示。
- 结果受模型版本和随机性影响，应在每次模型升级后重复运行。
- 建议在 CI 中保留离线 pytest，将在线评测设为手动或定时任务，避免每次提交产生 API 成本。

## 七、复跑方式

离线回归测试：

    python -m pytest -q

真实在线评测需要在当前进程设置 OPENAI_API_KEY：

    python evaluation/evaluate_live_llm.py

密钥不得写入测试报告、JSON 结果或 Git 仓库。
